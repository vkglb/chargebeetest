# Model Context: Chargebee-style Billing Platform

This file provides future AI assistants and developers with a comprehensive architectural summary of the subscription billing platform, avoiding the need to parse the entire codebase from scratch.

---

## 1. Project Overview & Multi-Tenancy

This project is a multi-tenant subscription billing platform modeled after Chargebee. It allows merchants (businesses) to onboard, connect payment gateways, define products/prices, create customers, and run subscription cycles.

- **Multi-Tenant Isolation**: The system partitions data using a merchant ID. Each action (e.g., creating a product, charging a customer) is authenticated and isolated under the merchant's profile.
- **Dynamic Gateways**: Supports multi-gateway setups (Stripe, Razorpay, Braintree, PayPal, and Sandbox). Credentials for these gateways are stored at-rest using AES-256-GCM encryption.

---

## 2. Tech Stack

### Backend
- **Language/Runtime**: Go 1.25
- **HTTP Routing**: Chi Router (`internal/server/`)
- **Database & Persistence**: PostgreSQL 15+, `pgx` (driver), `sqlc` (type-safe SQL compiler)
- **Migrations**: `goose` (`migrations/`)
- **Authentication**: JWT/bcrypt (`internal/auth/`)
- **Third-Party Payments**: `stripe-go` and custom payment gateway clients.

### Frontend
- **Framework/Runtime**: React 18, TypeScript, Vite
- **Routing**: React Router
- **CSS Styling**: Vanilla CSS / Responsive Layouts
- **Proxying**: Vite proxies API calls (`/v1` prefix) to the backend on `:8080`.

## 3. Directory Layout & Key Components

### Root Structure
- `backend/` - The Go application code.
- `frontend/` - The React SPA dashboard.
- `research/` - Notes on Chargebee architecture, Stripe flows, and migrations.

### Backend Layout (`backend/`)
- **`cmd/api/main.go`**: System entrypoint. Sets up dependencies, starts the HTTP server, and boots the background billing scheduler.
- **`internal/billing/`**: Contains period math, proration, dunning rules, invoice generation, and the core billing engine loop.
- **`internal/gateway/`**: Holds the `PaymentGateway` interface and provider implementations (`stripe`, `razorpay`, `braintree`, `paypal`, `sandbox`).
- **`internal/db/` & `internal/db/sqlc/`**: PostgreSQL pool helpers and generated database access layers.
- **`internal/server/`**: API routes, middleware, request/response models, and controllers.
- **`queries/`**: The `.sql` files parsed by `sqlc` to generate database code.
- **`migrations/`**: SQL migration scripts executed via `goose`.

### Frontend Layout (`frontend/src/`)
- **`api/client.ts`**: The typed client wrapper handling API requests, headers, JWT tokens, and shared backend type schemas.
- **`auth/AuthContext.tsx`**: State management for user credentials, log in, sign up, and persistent session state.
- **`components/`**: Layout panels (Sidebar, Protected routes).
- **`pages/`**: View layouts for Sites, Products, Product Details, Customers, Subscriptions, Coupons, Invoices, Transactions, and Overview.

## 4. Key Workflows & System Mechanics

### A. Core Billing Scheduler
The background scheduler is instantiated at boot time (`backend/cmd/api/main.go`):
1. Ticks every minute (`time.Minute`).
2. Calls the billing engine to query database records for due subscriptions.
3. Calculates period bounds, checks for proration, and generates invoice items.
4. Attempts an off-session charge through the merchant's active gateway (using idempotency keys).
5. On **success**: Advances the subscription period.
6. On **failure**: Transitions status to `past_due` and schedules dunning events (retries on Days 1, 3, and 5).

### B. Event Dispatching
Events (such as invoice created, charge failed, subscription updated) are dispatched via a dual publisher:
- **`webhooks.Dispatcher`**: Sends outbound HTTP webhooks to external URLs configured by the merchant.
- **`realtime.Hub`**: Broadcasts event notices over WebSockets directly to active frontend dashboards for live updates.

### C. Gateway Credential Security
Gateway credentials (like Stripe secret keys) are encrypted before writing to PostgreSQL via `internal/crypto`. The platform utilizes AES-256-GCM. When the engine processes a payment, it decrypts credentials on-the-fly using a key provided via configuration (`CredentialsEncKey`).

## 5. Development & Running the App

### Backend Setup
1. Copy `.env.example` to `.env` and fill variables.
2. Initialize tools: `make tools` (installs `goose` and `sqlc`).
3. Migrate DB: `make db-up` (runs Docker PG) or prepare local PG database, then run `make migrate-up`.
4. Build/Run API and Scheduler: `make run` (starts server on port `:8080`).

### Frontend Setup
1. Enter `frontend/` directory.
2. Install packages: `npm install`.
3. Start development server: `npm run dev` (defaults to port `:5173`).

### Regenerating DB Code
If you modify `.sql` query files in `backend/queries/` or DB schemas in `backend/migrations/`, run:
```bash
make sqlc
```
This regenerates `backend/internal/db/sqlc/` automatically.
